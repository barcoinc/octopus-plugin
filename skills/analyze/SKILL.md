# ターゲット分析フロー

## 目的
ターゲットの信念・矛盾を分析してtarget-hypothesisを生成する。
ここが台本の品質を一番左右する。分析なしで台本を書くことは禁止。

## 前提条件
`get_config(project_id)` でconfigが存在すること。なければsetupを先に実行。

## 分析エンジン（6ステップ）

### 1. パターン判定
- **A: 過去の自分救済型** — 自分が同じ問題を抱えていた → Before Storyが鏡になる
- **B: プロセス導出型** — 専門性で商品開発 → お客さんのBefore Storyを借りる
- **C: ハイブリッド型**

### 2. 信念・価値観の抽出
**上位にフォーカス:**
- 信念:「〇〇であるべき」「〇〇は無理」
- 価値観: 何を大切に / 何が許せないか
- アイデンティティ:「自分は〇〇な人間」

**下位は手がかり（出力の中心にしない）:**
- 環境・行動・能力 → これらの裏にある「信念」を読む

### 3. 矛盾の検出
矛盾の裏にある「真の信念」を見つける。指摘するためじゃない。
- 言ってること vs やってること
- 求めてること vs 避けてること
- 自己認識 vs 周囲の認識

### 4. 象徴的エピソード（取れた場合のみ）
感情強度が高い、繰り返し言及される、その後の選択を説明できるエピソード。

### 5. 結晶化の核
「抽象化」ではなく「重ね合わせ」。具体を保ったまま輪郭を濃くする。
- 悪い例: 「自信がない人」→ 薄すぎ
- 良い例: 「"もう失敗したくない"が口癖で、情報だけは集めるが一歩踏み出せない」→ 設計に直結

### 6. 確信度: 高（複数ソース裏付け） / 中（1ソース直接表現） / 低（推論）

## 出力
分析結果をユーザーに提示: 「ターゲット、こういう人ですよね？」
- 承認 → `save_analysis()` で保存
- 修正 → 更新して再確認

## 重要な原則
- 「属性」で止まらない。「信念」まで掘る
- 直接聞いても核心は出てこない。矛盾と象徴的エピソードがトリガー
- 1回で完璧にならなくていい。セミナーの反応で検証・更新する
